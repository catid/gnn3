# Round 13 Deployment Study

## Decision

Wave G was **not run as a new deployment panel**.

## Why

The deployment panel was conditional on a genuinely promoted branch from Waves D through F:

- a composite dispatcher that beats the best single branch
- or a conservative student that stays safe on Tier 2
- or a new single branch that materially improves the robust frontier

Round 13 produced none of those beyond the already accepted branchwise-max result.

## What Changed Instead

Wave A and Wave B narrowed the deployment story:

- archived frontier diversity was not robust
- only `prototype_branchwise_max_negative_cleanup_support_agree_mix_hybrid` survived the rerun robustness gate as a promoted branch
- the hierarchical dispatcher failed
- the conservative student retry was closed without training

So a new deployment comparison would have reduced to:

- `multiheavy`
- existing round-11/round-12 references
- the already accepted branchwise-max prototype branch

That is not a new round-13 deployment panel; it is the same deployment question with a single robust prototype candidate.

## Round-13 Deployment Conclusion

Do not promote a new deployment bundle from round 13.

Carry forward only:

- `multiheavy` as the default policy
- `prototype_branchwise_max_negative_cleanup_support_agree_mix_hybrid` as the only robust prototype-side candidate for any future deployment comparison

If a later round opens a deployment panel, it should compare those two directly at matched budgets, not reopen the failed composite or student paths.
